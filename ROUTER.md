# ROUTER.md — Логика роутинга

> Как LLM решает, какие скиллы загружать под конкретный запрос.
> Эта логика вшита в `PROJECT_INSTRUCTIONS.md` и применяется в каждой сессии.

---

## Уровни приоритета (всегда, в этом порядке)

1. **`/people/<active-user>/`** — личный контекст конкретного дизайнера. Грузится **первым** в каждой сессии.
2. **`/team/`** — командный контекст. Грузится **первым** в каждой сессии.
3. **`/skills/meta/`** — кросс-доменные скиллы (design-critique, accessibility, handoff). Грузятся, когда в запросе есть соответствующий триггер.
4. **`/skills/<domain>/`** — основной доменный скилл под суть запроса.
5. **`/mirrors/`** — зеркала внешних скиллов как авторитетный источник (Anthropic и т.п.).
6. **`INDEX.md`** (long tail внешних ссылок) — fallback, когда ничего из локального не покрывает.

**Принципы:**
- **Локальное побеждает внешнее.** Свои скиллы перебивают зеркала и внешние ссылки.
- **Личное побеждает командное** на уровне стиля и подхода (см. нюансы ниже).
- **Командное побеждает личное** на уровне обязательных стандартов.

---

## Алгоритм решения (по шагам)

```
ШАГ 1A. Загрузи /team/ контекст:
        - team/profile.md
        - team/design-principles.md
        - team/current-projects.md  (только если запрос про проект)
        - team/brand-tone.md         (только если запрос про user-facing copy)

ШАГ 1B. Загрузи /people/<active-user>/ контекст:
        - people/<active-user>/profile.md
        - people/<active-user>/tone-of-voice.md
        - people/<active-user>/principles.md
        
        <active-user> — захардкожен в PROJECT_INSTRUCTIONS у каждого
        члена команды через `<REPLACE_WITH_YOUR_USERNAME>`.

ШАГ 2. Определи домен(ы) запроса по таблице триггеров (см. ниже).
        Может быть 1, 2, или 3 домена одновременно — это нормально.

ШАГ 3. Подключи META-скиллы, если в запросе есть:
        • ревью / критика / фидбек        → /skills/meta/design-critique/
        • a11y / доступность / контраст   → /skills/meta/accessibility/
        • handoff / спека / разработчик   → /skills/meta/handoff/

ШАГ 4. Загрузи корневой SKILL.md основного домена.
        Если в нём указан подскилл — загрузи его тоже.

ШАГ 5. Проверь `depends_on` — догрузи зависимости.

ШАГ 6. Если ни один локальный скилл не покрывает запрос — загляни в INDEX.md
        и подгрузи внешний.

ШАГ 7. Сформируй ответ. В конце ответа кратко перечисли,
        какие скиллы были использованы.
```

## Конфликты и приоритезация

### Случай 1. Триггерится несколько скиллов одновременно

**Пример:** «Сделай ревью текста на онбординге с точки зрения a11y».

**Решение:** загружай **все три** скилла — `ux-writing`, `meta/design-critique`, `meta/accessibility`. Применяй последовательно:
1. `ux-writing` задаёт критерии «правильного» текста.
2. `meta/design-critique` структурирует ревью.
3. `meta/accessibility` проверяет a11y-аспекты.

В ответе явно укажи, какие критерии применяешь и из какого скилла.

### Случай 2. Личные принципы vs. командные принципы

**Пример:** в `/team/design-principles.md` написано «всегда используем sentence case в CTA», а в `/people/anna/principles.md` написано «использую Title Case в важных CTA».

**Решение:** **личное побеждает в выборе стиля.** Это её право. Но если бы конфликт был «команда требует WCAG AA, а у Анны написано "не считаю важным"» — **командное побеждает**, потому что это обязательный стандарт.

**Правило:**
- **Обязательные стандарты** (a11y, compliance, безопасность) — команда побеждает.
- **Стиль / подход / методология** — личное побеждает.
- **В спорных случаях** — явно укажи конфликт в ответе и предложи компромисс.

### Случай 3. Личный tone-of-voice vs. бренд-голос продукта

**Пример:** Анна пишет коротко и иронично, а бренд продукта — формально-нейтральный.

**Решение:** зависит от **получателя текста:**
- Текст для пользователя (microcopy, ошибки, маркетинг) → побеждает `team/brand-tone.md`.
- Внутренняя коммуникация (комментарии, Slack, документация) → побеждает `people/<name>/tone-of-voice.md`.
- Внутренний документ о продукте → личный стиль, но термины и формулировки бренда из `brand-tone.md`.

### Случай 4. Зеркало Anthropic vs. собственный скилл

**Решение:** свой выигрывает (адаптирован под команду), но в ответе сошлись на оба источника, если они касаются одной темы. Это полезно для самоконтроля.

### Случай 5. Ничего не триггерится явно

**Решение:**
1. Сначала попробуй сопоставить запрос с цепочками (см. ниже) — может быть, это end-to-end задача.
2. Если нет — запроси у пользователя уточнение **одной короткой репликой** (не списком вопросов).
3. В крайнем случае примени `/skills/meta/design-critique/` как универсальный фолбэк.

## Цепочки скиллов (chains)

Типичные end-to-end последовательности. Если запрос «охватывающий» — прогоняй через цепочку с промежуточными выводами.

| Цепочка | Когда применять |
|---|---|
| `ux-research` → `research-synthesis` → `design-critique` | «Провели интервью, помоги превратить инсайты в фидбек на текущий макет» |
| `product-analytics` → `ux-writing` → `design-critique` | «Кнопка не конвертит — перепиши и оцени» |
| `ux-research` → `product-design` → `design-systems` | «Спроектировать новую фичу по результатам ресёрча» |
| `branding` → `graphic-design` → `ux-writing` | «Сделать лендинг под новый бренд» |
| `product-design` → `meta/accessibility` → `meta/handoff` | «Финализировать экран перед передачей разработчикам» |
| `design-systems` → `meta/design-critique` → `meta/handoff` | «Аудит компонента в DS и подготовка к передаче» |

Цепочки не обязательны — это рекомендованные паттерны.

## Поведение «по умолчанию»

- Тон ответа — из `people/<active-user>/tone-of-voice.md` для внутренних коммуникаций, из `team/brand-tone.md` для текстов на пользователя.
- Принципы — слой `people/` + слой `team/`. Личное в стиле и подходе, командное в обязательных стандартах.
- Если запрос упоминает текущий проект — обязательно прочитай `team/current-projects.md`.
- Если запрос про незнакомую команде индустрию (нет в `team/profile.md`) — явно скажи «вне основного опыта команды» и попроси уточнить.

---

## Что роутер НЕ делает

- **Не грузит всё подряд.** Засоряет контекст и снижает качество.
- **Не игнорирует `/team/` и `/people/`.** Это база каждой сессии.
- **Не лезет в `INDEX.md` без необходимости.** Внешние скиллы — только когда локальные не покрывают.
- **Не комбинирует противоречащие источники молча.** Если зеркало говорит одно, а свой скилл другое — явно укажи в ответе.
- **Не пересказывает скиллы.** Применяй их к запросу, не цитируй.
- **Не путает слои.** `team/brand-tone.md` — для продукта, `people/<name>/tone-of-voice.md` — для дизайнера.

## Прозрачность для отладки

В конце каждого ответа кратко перечисли использованные скиллы:

```
Использованные скиллы:
• team/profile, design-principles
• people/anna/tone-of-voice, principles
• skills/ux-writing/microcopy v1.2.0
• mirrors/anthropic/design-critique
```

Это помогает увидеть, что роутер сработал верно, и быстро находить «дырявые» места.
