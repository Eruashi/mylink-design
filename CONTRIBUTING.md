# CONTRIBUTING.md — Как развивать репозиторий

> Правила для тебя самого (и для будущих контрибьюторов, если репозиторий станет публичным).

---

## 1. Добавить новый скилл

### Шаги

1. Определи **домен** (один из 9): `product-design`, `design-systems`, `ux-writing`, `ux-research`, `product-analytics`, `ux-ui-theory`, `graphic-design`, `branding`, `meta`.
2. Создай директорию `skills/<domain>/<skill-name>/` в kebab-case.
3. Скопируй болванку из `templates/skill-skeleton.md` в `SKILL.md`.
4. Заполни frontmatter согласно `SCHEMA.md`.
5. Заполни тело по обязательным секциям: Когда / Что / Контекст / Метод / Формат / Анти-паттерны / Примеры / Связи / Источники.
6. Если `SKILL.md` приближается к 300 строкам — выноси глубокие материалы в `references/`.
7. Добавь минимум один тестовый запрос в `evals/test-prompts.md` и проверь, что роутер триггерит скилл.
8. Запиши изменение в `CHANGELOG.md`.
9. Обнови `INDEX.md`, если меняется навигация.

### Чеклист перед коммитом

- [ ] Frontmatter полный (все поля из `SCHEMA.md`).
- [ ] `description` на английском, «pushy» — явно перечислены триггеры.
- [ ] Все обязательные секции тела заполнены.
- [ ] Связи (`Predecessor` / `Successor` / `Conflicts`) указаны.
- [ ] `<300` строк или вынесено в `references/`.
- [ ] Запись в `CHANGELOG.md`.
- [ ] Тестовый запрос в `evals/test-prompts.md` проходит.

## 2. Обновить существующий скилл

### Bumping версии

- **Patch** (1.0.0 → 1.0.1) — опечатки, мелкие уточнения, добавление примеров без изменения метода.
- **Minor** (1.0.0 → 1.1.0) — новая секция, новый шаг в методе, расширение функциональности без breaking change.
- **Major** (1.0.0 → 2.0.0) — изменился формат вывода, методика, структура тела. Запросы, которые работали раньше, могут работать иначе.

### Шаги

1. Внеси изменения.
2. Обнови `version` и `last_updated` во frontmatter.
3. Если изменения breaking — обнови `depends_on` зависимых скиллов, если они полагались на старое поведение.
4. Запиши в `CHANGELOG.md` формата `X.Y.Z → X.Y.Z` с кратким описанием.
5. Если описание триггеров (`description`) меняется — проверь, что роутер всё ещё подцепляет скилл. Прогони тестовые запросы.

## 3. Добавить зеркало внешнего скилла

Используй только для критичных и стабильных внешних скиллов. Остальное — в `INDEX.md`.

### Шаги

1. Создай директорию `mirrors/<source>/<skill-name>/`.
2. Скопируй оригинальный `SKILL.md` один-в-один. **Не редактируй.**
3. В frontmatter обнови `mirrors: <url-оригинала>`.
4. Создай `mirrors/<source>/<skill-name>/ATTRIBUTION.md`:
   ```markdown
   # Attribution

   - **Source:** <url>
   - **Snapshot commit:** <hash или дата>
   - **Snapshot date:** YYYY-MM-DD
   - **License:** <license>
   - **Author:** <author>
   ```
5. Свои дополнения / комментарии / адаптации — в отдельном файле `EXTENSIONS.md` рядом с `SKILL.md`. Туда же — твои примеры, реальные кейсы, исправления неточностей.
6. Запиши в `CHANGELOG.md`.

### Обновление зеркала

1. Сравни с актуальной версией оригинала (`diff`).
2. Если в оригинале breaking change — оцени, не сломаются ли твои `EXTENSIONS.md`.
3. Перезапиши `SKILL.md` новой версией, обнови `snapshot date` в `ATTRIBUTION.md`.
4. Запиши в `CHANGELOG.md`.

## 4. Добавить template или framework

### Templates (`/templates/`)

Это заготовки готовых deliverables: research-report, design-spec, ab-test-plan и т.п.

- Один файл = один тип deliverable.
- В начале файла — комментарий с описанием: когда использовать, какие скиллы дополняют.
- Поля для заполнения — в `[квадратных скобках]` или `_подчёркиваниях_`.
- Запись в `CHANGELOG.md`.

### Frameworks (`/frameworks/`)

Это краткие справочники по конкретным фреймворкам: Nielsen heuristics, JTBD, HEART metrics и т.п.

- Один файл = один фреймворк.
- Объём: 1–3 экрана, не учебник.
- Цель — чтобы LLM могла быстро применить фреймворк, не «вспоминая по памяти».
- Запись в `CHANGELOG.md`.

## 5. Обновить `/personal/`

Это твои файлы — обновляй когда нужно. Особо:

- При смене работы / проекта — обнови `current-projects.md`.
- При изменении стиля / роли — обнови `tone-of-voice.md` и `profile.md`.
- В `CHANGELOG.md` указывай минимально (`personal/* — обновлены контекстные данные`), без деталей, чтобы не палить персональное в публичных commit history.

---

## 6. Commit messages

Формат:

```
<scope>: <краткое описание>

<тело при необходимости>
```

`<scope>` — путь до изменённого файла или директории:

- `schema: уточнено правило о длине SKILL.md`
- `router: добавлена цепочка branding → graphic-design → ux-writing`
- `skills/ux-writing/microcopy: добавлена секция про error states`
- `mirrors/anthropic/design-critique: обновлено зеркало до 2026-05-12`
- `personal: обновлены current-projects`

---

## 7. Когда сомневаешься

- В формате — смотри `SCHEMA.md`.
- В подключении — смотри `ROUTER.md`.
- В роли скилла — смотри соседей в том же домене.
- В принципе скилла — смотри `/personal/design-principles.md`.

Если всё ещё не уверен — оставь `status: draft` во frontmatter. Это сигнал, что скилл ещё не готов к продакшен-использованию, и роутер может его игнорировать.
