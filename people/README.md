# /people/ — Личный контекст дизайнеров

Эта папка содержит **личный** контекст каждого дизайнера команды. Командное — в `/team/`.

---

## Зачем это нужно

Командный контекст (`/team/profile.md`, `/team/design-principles.md`, `/team/brand-tone.md`) описывает команду как единицу: компания, индустрия, общие принципы, бренд-голос продукта.

Но у каждого дизайнера есть **свой** опыт, стиль письма и принципы, которые делают его работу узнаваемой:

- **Алекс** пишет коротко, без лишних слов, использует ироничные комментарии на ревью.
- **Мария** даёт развёрнутый фидбек с эмпатией, разворачивает каждый пункт.
- **Юра** специализируется на анимации и moation design, у него своя теория mind-set.

Если LLM работает «усреднённо» по командному контексту, она теряет эту индивидуальность. Поэтому каждый дизайнер заводит свою папку `/people/<имя>/` и кладёт туда личные файлы.

---

## Что в каждой папке

```
/people/<имя>/
├── profile.md         # личный опыт, специализация, anti-cred
├── tone-of-voice.md   # как ты пишешь в Slack, в комментариях, в документации
└── principles.md      # личные принципы (часто пересекаются с командными, но не всегда)
```

## Onboarding нового члена команды

### Шаг 1. Создай свою папку

В корне `/people/` создай папку с твоим именем — желательно kebab-case на латинице:

```
/people/alex-ivanov/
/people/maria-k/
/people/yura/
```

### Шаг 2. Скопируй шаблон

В `/people/_template/` лежат болванки трёх файлов. Скопируй их в свою папку:

```
/people/alex-ivanov/profile.md
/people/alex-ivanov/tone-of-voice.md
/people/alex-ivanov/principles.md
```

### Шаг 3. Заполни

Заполни файлы под себя. Приоритет — `tone-of-voice.md`: **обязательно вставь 2–3 реальных примера своих текстов**, это самый ценный сигнал для LLM.

В остальных файлах смело пропускай поля, которые ещё не знаешь, как заполнить. Лучше пусто, чем общие фразы.

### Шаг 4. Настрой свой Claude Project

Открой свой Claude Project → `Settings → Custom instructions`.

Скопируй туда содержимое `/PROJECT_INSTRUCTIONS.md` из корня репозитория. В нём есть строка:

```
**Active user:** <REPLACE_WITH_YOUR_USERNAME>
```

Замени `<REPLACE_WITH_YOUR_USERNAME>` на имя своей папки (например, `alex-ivanov`). Сохрани.

Это критично — без этого Claude не знает, какой `/people/<имя>/` грузить.

### Шаг 5. Проверь

Напиши Claude в этом проекте простое сообщение:

> «Какой у меня стиль письма?»

Если всё настроено правильно — Claude в ответе сошлётся на твой `/people/<имя>/tone-of-voice.md` и опишет твой стиль. В конце ответа должен быть блок «Использованные скиллы», и среди них — `people/<твоё имя>/*`.

Если Claude не нашёл твой файл — проверь:
- Username в PROJECT_INSTRUCTIONS совпадает с именем папки.
- Папка закоммичена в основную ветку (Claude не видит локальные изменения).
- В `tone-of-voice.md` есть хоть какое-то содержание (Claude может не подгружать пустой файл).

## Что меняется в `/people/<имя>/` со временем

- **При смене проекта или фокуса** — обнови `profile.md` (специализация может сместиться).
- **При изменении стиля письма** — обнови `tone-of-voice.md`.
- **При принципиальных дизайн-инсайтах** — обнови `principles.md`.
- **Раз в полгода** — пройдись по всем трём файлам, сверь с реальностью.

---

## Конфликт «личное vs командное»

Иногда личный принцип в `/people/<имя>/principles.md` противоречит командному в `/team/design-principles.md`. Правила:

- **Личное побеждает** на уровне выбора стиля, подхода, формулировки.
- **Командное побеждает** на уровне обязательных стандартов (a11y, compliance, безопасность).
- **В спорных случаях** LLM явно отметит конфликт в ответе и предложит компромисс.

Если ты обнаруживаешь, что регулярно споришь с командным принципом — это сигнал поднять обсуждение в команде и обновить `/team/design-principles.md`, а не молча писать в свой personal.

---

## Приватность

Файлы в `/people/<имя>/` лежат в общем репозитории и видны всем участникам. Не клади туда:

- Конфиденциальную информацию о клиентах
- Зарплатные цифры
- NDA-данные
- Что-то, что ты не хочешь, чтобы коллега увидел в commit history

Если репозиторий когда-то станет публичным — вычисти `/people/` заранее.

---

## Уход из команды

Когда дизайнер уходит:

1. Папка `/people/<имя>/` **не удаляется сразу** — на ней могут быть отсылки в архиве проектов.
2. Через 1 месяц после ухода — папка переносится в `/people/_archive/<имя>/`.
3. PROJECT_INSTRUCTIONS другие дизайнеры обновляют, чтобы не ссылаться случайно.
