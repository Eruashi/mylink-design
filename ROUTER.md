# ROUTER.md — Логика роутинга

> Как LLM решает, какие скиллы загружать под конкретный запрос.
> Эта логика вшита в `PROJECT_INSTRUCTIONS.md` и применяется в каждой сессии.

---

## Уровни приоритета (всегда, в этом порядке)

1. **`/personal/`** — твой контекст. Грузится **первым** в каждой сессии. Без исключений.
2. **`/skills/meta/`** — кросс-доменные скиллы (design-critique, accessibility, handoff). Грузятся, когда в запросе есть соответствующий триггер.
3. **`/skills/<domain>/`** — основной доменный скилл под суть запроса.
4. **`/mirrors/`** — зеркала внешних скиллов как авторитетный источник (Anthropic и т.п.).
5. **`INDEX.md`** (long tail внешних ссылок) — fallback, когда ничего из локального не покрывает.

Принцип: **локальное побеждает внешнее, персональное побеждает любое общее.**

---

## Алгоритм решения (по шагам)

```
ШАГ 1. Загрузи /personal/profile.md, tone-of-voice.md, design-principles.md.
        current-projects.md — только если в запросе есть отсылка
        к «текущему проекту», «нашему продукту», «моей фиче».

ШАГ 2. Определи домен(ы) запроса по таблице триггеров (см. ниже).
        Может быть 1, 2, или 3 домена одновременно — это нормально.

ШАГ 3. Подключи META-скиллы, если в запросе есть:
        • ревью / критика / фидбек        → /skills/meta/design-critique/
        • a11y / доступность / контраст   → /skills/meta/accessibility/
        • handoff / спека / разработчик   → /skills/meta/handoff/

ШАГ 4. Загрузи корневой SKILL.md основного домена.
        Если в нём указан подскилл (например, /skills/ux-writing/microcopy/) — загрузи его тоже.

ШАГ 5. Проверь `depends_on` — догрузи зависимости.

ШАГ 6. Если ни один локальный скилл не покрывает запрос — загляни в INDEX.md
        и подгрузи внешний.

ШАГ 7. Сформируй ответ. В конце ответа кратко перечисли,
        какие скиллы были использованы (для прозрачности и отладки).
```

## Конфликты и приоритезация

### Случай 1. Триггерится несколько скиллов одновременно

**Пример:** «Сделай ревью текста на онбординге с точки зрения a11y».

**Решение:** загружай **все три** скилла — `ux-writing`, `meta/design-critique`, `meta/accessibility`. Применяй последовательно:
1. `ux-writing` задаёт критерии «правильного» текста.
2. `meta/design-critique` структурирует ревью.
3. `meta/accessibility` проверяет a11y-аспекты.

В ответе явно укажи, какие критерии применяешь и из какого скилла.

### Случай 2. Персональные принципы vs. рекомендация скилла

**Пример:** скилл говорит «избегай восклицательных знаков», но `tone-of-voice.md` разрешает их в успешных состояниях.

**Решение:** **`/personal/` всегда выигрывает.** Скилл — это общая практика, твой контекст — конкретный. Если конфликт серьёзный (например, личный принцип противоречит WCAG) — явно отметь это в ответе и предложи компромисс.

### Случай 3. Зеркало Anthropic vs. твой собственный скилл

**Решение:** твой выигрывает (он адаптирован под тебя), но в ответе сошлись на оба источника, если они касаются одной темы. Это полезно для самоконтроля — иногда оригинал лучше.

### Случай 4. Ничего не триггерится явно

**Решение:**
1. Сначала попробуй сопоставить запрос с цепочками (см. ниже) — может быть, это end-to-end задача.
2. Если нет — запроси у пользователя уточнение **одной короткой репликой** (не списком вопросов).
3. В крайнем случае примени `/skills/meta/design-critique/` как универсальный фолбэк для любых дизайн-задач.

## Цепочки скиллов (chains)

Типичные end-to-end последовательности. Если запрос «охватывающий» — прогоняй через цепочку с промежуточными выводами.

| Цепочка | Когда применять |
|---|---|
| `ux-research` → `research-synthesis` → `design-critique` | «Провели интервью, помоги превратить инсайты в фидбек на текущий макет» |
| `product-analytics` → `ux-writing` → `design-critique` | «Кнопка не конвертит — перепиши и оцени» |
| `ux-research` → `product-design` → `design-systems` | «Спроектировать новую фичу по результатам ресёрча» |
| `branding` → `graphic-design` → `ux-writing` | «Сделать лендинг под новый бренд» |
| `product-design` → `meta/accessibility` → `meta/handoff` | «Финализировать экран перед передачей разработчикам» |
| `design-systems` → `meta/design-critique` → `meta/handoff` | «Аудит компонента в DS и подготовка к передаче» |

Цепочки не обязательны — это рекомендованные паттерны. Если запрос узкий, применяй один скилл.

## Поведение «по умолчанию»

- Тон ответа — из `personal/tone-of-voice.md`.
- Принципы (например, «не давай советов без контекста проекта») — из `personal/design-principles.md`.
- Если запрос упоминает текущий проект — обязательно прочитай `personal/current-projects.md`.
- Если запрос про незнакомую тебе индустрию из `profile.md` — явно скажи «вне моего основного опыта» и попроси уточнить контекст.

---

## Что роутер НЕ делает

- **Не грузит всё подряд.** Засоряет контекст и снижает качество. Грузи только триггернувшееся.
- **Не игнорирует `/personal/`.** Это база каждой сессии.
- **Не лезет в `INDEX.md` без необходимости.** Внешние скиллы — только когда локальные не покрывают.
- **Не комбинирует противоречащие источники молча.** Если зеркало говорит одно, а твой скилл другое — явно укажи в ответе.
- **Не пересказывает скиллы.** Применяй их к запросу, не цитируй.

## Прозрачность для отладки

В конце каждого ответа кратко перечисли использованные скиллы:

```
Использованные скиллы:
• personal/profile, tone-of-voice
• skills/ux-writing/microcopy v1.2.0
• skills/meta/design-critique v1.0.0
```

Это поможет тебе видеть, что роутер сработал верно, и быстро находить «дырявые» места, где скиллы недотриггериваются.
