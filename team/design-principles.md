## Базовые принципы команды

> Принципы выведены из практики — это то, что Екатерина и команда регулярно проверяют на ревью. Применимо ко всем дизайнерам и проектам.

1. **Все состояния всегда.** Любой компонент или экран отдаётся с проработанными состояниями: default, hover, active, focus, disabled, loading, error, empty. Отсутствие хотя бы одного состояния — повод вернуть работу на доработку.
2. **Мобильная версия — не «потом».** Для каждой задачи продумывается, как решение будет жить на мобильном экране (не только в продуктовом приложении на Flutter, но и в адаптиве web-интерфейса). «Я сделаю мобилку позже» — не работает.
3. **Только наши токены и стили.** В макетах используются токены из дизайн-системы, не magic numbers и не цвета из чужих палитр. Magic number в макете = запах.
4. **Локальные компоненты — если их нет в DS, создавай.** Если паттерн повторяется, но в DS его нет — нельзя «нарисовать поверх». Создаётся локальный компонент с правильной структурой. Это касается даже одноразовых вещей: завтра кто-то их переиспользует.
5. **Flow chart обязателен для нелинейных сценариев.** Если в задаче больше одного экрана и есть ветвления — рисуется flow chart рядом с макетами. Без него разработчик и сам дизайнер через две недели уже не вспомнят логику.
6. **Комментарии для разработчиков — часть деливерабла.** Не «что я нарисовал», а «что должно произойти при таком-то условии», «почему именно так», «что критично соблюсти». Комментарии оставляются прямо в Figma.
7. **Контекст задачи фиксируется дизайнером.** Если PM объяснил задачу устно — дизайнер сам записывает понимание (в Confluence или комментариях к задаче). Принимаемые решения, которые приняли вне дизайнера — обязательно уточняются.
8. **Решение должно упрощать пользователю, а не команде.** Если фича сложнее с точки зрения пользователя, чем была раньше — это шаг назад, даже если бизнесу так удобнее.
9. **Консистентность с другими экранами продукта.** Похожие задачи на разных экранах решаются одинаково. Если делаешь иначе — должна быть явная причина, которая фиксируется в комментариях.

## Что для нас «хороший дизайн»

- Все интерактивные состояния проработаны и переданы разработке без вопросов
- Дизайн консистентен с остальным продуктом — компоненты, токены, паттерны
- Логика взаимодействий понятна без объяснений (либо объяснения зашиты прямо в макет: flow chart, комментарии)
- Решение упрощает жизнь пользователю, а не команде

## Что для нас «плохой дизайн»

- Макет без всех состояний
- Использование токенов или стилей не из нашей DS
- Локальные кастомные компоненты вместо тех, что в DS, без обоснования
- Отсутствие flow chart для нелинейных сценариев
- Отсутствие комментариев для разработчиков там, где они нужны
- Дизайнер не уточнил подробности задачи и не интересовался решениями, принятыми без него
- Непоследовательность с другими экранами продукта без объяснения причин

## Обязательные стандарты

> Без этого работа не отдаётся. Применимо ко всем дизайнерам и всем проектам.

### Accessibility

- [~] Контраст WCAG AA минимум для всех текстов (4.5:1 normal, 3:1 large)
- [~] Контраст UI элементов ≥ 3:1 (границы, focus ring)
- [~] Touch targets ≥ 44pt на mobile
- [x] Все интерактивные состояния спроектированы: default, hover, active, focus, disabled, loading, error
- [x] Focus order логичный, focus visible
- [x] Информация не передаётся ТОЛЬКО цветом
- [ ] Reduced motion / prefers-reduced-motion поддерживается
- [x] Empty states для всех data-driven экранов
- [x] Loading skeletons / shimmer для ожидания ≥ 500ms

### Локализация / интернационализация

- [x] +30% длины текста при переводе (русский/казахский) учитывается в макетах
- [ ] RTL-готовность — не применимо, продукт работает в Казахстане

### Дизайн-система

- [x] Используются токены, не magic numbers
- [~] Компоненты из DS, не локальные кастомные без обоснования
- [x] Все states компонентов согласованы с другими экранами

## Чего команда избегает

> Конкретные паттерны, антипаттерны, тренды. На что LLM должна реагировать предупреждением, если такое решение всплывёт в запросе.

### Dark patterns

Команда не использует манипулятивные паттерны:

- Ложные срочности и FOMO-таймеры
- Confirmshaming («Нет, я не хочу быть продуктивнее»)
- Скрытые согласия и предзаполненные чекбоксы на чувствительные действия
- Принуждение сотрудника к действиям через интерфейс (контекст HR-софта — особенно важно, чтобы инструмент не работал против пользователя в пользу работодателя)

LLM должна предупреждать, если предложение скатывается в эту сторону.

### Визуальные тренды, которые НЕ применяем

- Скевоморфизм
- Neumorphism
- Glassmorphism
- Перегруженный motion design (анимации ради анимаций)

**Почему:** не подходит к B2B HR-софту. Эти стили либо снижают читаемость, либо отвлекают от рабочих задач, либо устаревают слишком быстро для долгоживущего корпоративного продукта.

## Принципы работы с разработкой

### Что готовим до handoff

- Все состояния компонентов и экранов
- Комментарии к макетам или компонентам там, где нужно: что должно произойти, при каком условии, что критично соблюсти
- Flow chart для нелинейных сценариев
- Демонстрация анимации (через прототип или запись), если в задаче есть motion

### Формат деливерабла

Статичные макеты в Figma — основной формат. Полноценные прототипы не делаются. Если в задаче есть анимация — показывается отдельно, не интегрируется в основной макет.

### Чего НЕ ждём от разработки

Эти решения принимает дизайнер сам, не делегирует:

- Тексты состояний ошибок, успехов и пустых состояний
- Логику и параметры анимаций (длительность, easing, триггеры)
- Выбор иконок и визуальные детали

### Что делаем, если макет нельзя реализовать

Каждый случай индивидуален — приходим к компромиссу с разработкой. Главная стратегия — не доводить до таких ситуаций: проводим tech design review с frontend-командой **до** финализации макета, чтобы поймать невозможное на раннем этапе.
